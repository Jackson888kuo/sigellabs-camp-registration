## 問題摘要

當家長在 Tally 報名表單選擇「兩個以上營隊」並僅勾選 **2 個營隊** 時，後端 Make.com 自動化流程卻將該家長寫入 **6 個營隊**（疑似全部 6 個夏令營皆被處理），且寄送出的確認信也包含 6 個以上營隊的繳費卡片（含 6 個獨立的「前往繳費」按鈕）。

此 bug 會嚴重影響家長體驗、產生錯誤帳務資料，並可能讓家長誤付不該繳的營隊費用。

---

## 重現步驟

1. 開啟 Tally 報名表單
2. 填寫家長 / 孩子基本資料（測試案例使用：testing0944 / jksnkuo@gmail.com / MJackson）
3. 「團報人數」選擇：**3人團報**
4. 「您這次要報名幾個營隊？」選擇：**兩個以上營隊**
5. 「孩子要報名哪些營隊？」**僅勾選 2 個營隊**：
   - [STEAM] Attack用電攻擊！拆解生活中的電學營隊
   - [運算思維] Game Designer！小小遊戲設計師營隊
6. 送出表單
7. 觀察「報名追蹤表」(Google Sheets) 與家長收到的確認信

## 初步根因推測

| 假設 | 描述 | 證據強度 |
| --- | --- | --- |
| **H1：Iterator 來源錯誤** | Make.com 場景裡的 Iterator 直接套用「活動設定表」的全部 6 個營隊，而非家長在 Tally 中勾選的子集合 | 高（追蹤表剛好出現全部 6 個營隊） |
| **H2：缺少 Filter 過濾步驟** | Iterator 之後沒有設定「僅當營隊名稱 ∈ Tally 勾選清單」的 Filter | 高 |
| **H3：Tally 多選欄位解析失敗** | 模組未把 `孩子要報名哪些營隊？` 欄位當成 array，而是退回到預設清單 | 中（需檢查 Tally Webhook payload 解析） |
| **H4：方案B Iterator 架構在「兩個以上營隊」分支未實作 selection-aware 邏輯** | 根據 memory 中 Phase 5 設定表 + Phase 4 Iterator 架構，可能此分支沿用 fallback 全營隊清單 | 中-高 |

---

## 建議排查方向

1. **檢視 Make.com 場景的 Iterator 模組**：確認 `Array` 來源是否來自 Tally 欄位 `孩子要報名哪些營隊？` 而非「活動設定表」的全部 rows。
2. **加入 Filter / Router 過濾條件**：在 Iterator 後加上 `IF camp_name ∈ tally_selected_camps` 才往下寫入 Sheets / 組合 email。
3. **驗證 Tally Webhook payload**：用 Make.com 的執行歷史確認 `孩子要報名哪些營隊？` 欄位在 webhook 中的格式（string vs array），調整 split / parse 邏輯。
4. **比對「單一營隊」分支 vs 「兩個以上營隊」分支**：根據 memory，方案B 是 Iterator 架構，但「兩個以上營隊」分支可能存在 regression。
5. **新增 unit-style 測試案例**：勾選 1 / 2 / 3 / 4 / 5 / 6 個營隊各跑一次，比對追蹤表與 email 結果。
