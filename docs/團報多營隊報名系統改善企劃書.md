# 太陽實驗室 2026 夏令營團報系統改善企劃書

## 專案背景

目前太陽實驗室的團報流程為：Tally 表單 → Make.com 自動化 → HubSpot CRM + Google Sheets 追蹤表 + SendGrid 確認信。

當家長在一張報名單中為同一位小孩同時報名多個營隊時，現有系統出現以下問題：

1. **HubSpot**：`interested_camp` 欄位僅寫入第一個營隊，其餘被忽略
2. **確認信**：僅顯示第一個營隊的付款按鈕（已修正）
3. **追蹤報名表**：無法正確呈現多營隊報名
4. **Tally Thank you 頁面**：多選時跳轉邏輯異常（已修正方向）

---

## 現有系統架構分析

### HubSpot CRM 現有結構

**Contact 層級（家長/學生）：**

| 欄位名稱 | 類型 | 用途 |
|---|---|---|
| `interested_camp` | enumeration (19 選項) | 意願登記的營隊類型（無日期梯次） |
| `child_1_course_attended` | enumeration (221 選項) | 實際上課的具體梯次（含日期地點） |
| `child_1_trial_course` | enumeration | 體驗課紀錄 |
| `child_1_date_of_birth` | string | 孩子生日 |
| `child_2_*`, `child_3_*` | 同上 | 第二、三個孩子的資料 |

**Deal 層級（訂單/報名）：**

| Pipeline | 用途 |
|---|---|
| Ecommerce Pipeline | 官網直接購買（Checkout Abandoned → Payment Pending → On hold → Processing → Completed → Refunded） |
| Offline | 線下報名（Inquiry → Trial → Registered → Closed won → Closed lost） |

**現有 Deal 自訂欄位：**

| 欄位名稱 | Label | 說明 |
|---|---|---|
| `child_1_deal_course` | Child1_課程名稱 | 課程名稱（可用 `;` 分隔多值） |
| `child_1_deal_class` | Child 1_班級 | 班級/梯次 |
| `child_1_deal_year` | Child 1_課程日期（年） | 課程年份 |
| `child_1_name` | — | 孩子姓名 |

**重要發現：** 現有的 Ecommerce Deal 中已經存在 `child_1_deal_course` 以分號分隔多營隊的紀錄（例如：`2026 旅程製造所-[台北] 2/2-2/6;2026 重返案發現場-[台北] 8/3-8/7`）。這表示 HubSpot 本身已有多營隊的資料模型，但目前團報流程尚未接入。

## 推薦方案：「一個營隊 = 一筆 Deal」Iterator 架構

### 設計原則

- **一份報名單、一位家長、一位小孩**：對應一個 Contact
- **每個報名的營隊**：對應一筆獨立的 Deal，關聯到該 Contact
- **追蹤報名表**：自然達成「一列 = 一學生 + 一營隊」
- **確認信**：一封信包含所有營隊的付款按鈕

### 3. Make.com 自動化流程

#### 3-1. 現有流程（修改前）

```
Tally Webhook → Create/Update Contact → Send Email (SendGrid)
```

#### 3-2. 新流程（修改後）

```
Tally Webhook
    │
    ├─ 1. Create/Update Contact
    │      - 寫入家長資訊、孩子資訊
    │      - interested_camp = 所有選取營隊（用 ; 串接）
    │
    ├─ 2. Iterator（遍歷每個選取的營隊）
    │      │
    │      ├─ 2a. Create Deal
    │      │      - pipeline: 2026 夏令營團報
    │      │      - dealstage: registered_unpaid
    │      │      - dealname: 團報_{孩子姓名}_{營隊簡稱}
    │      │      - child_1_deal_course: 該營隊完整名稱
    │      │      - child_1_name: 孩子姓名
    │      │      - amount: 營隊費用
    │      │
    │      ├─ 2b. Associate Deal ↔ Contact
    │      │
    │      └─ 2c. Add Row to Google Sheets（追蹤報名表）
    │             - 一列 = 一學生 + 一營隊
    │
    └─ 3. Send Email (SendGrid)
           - 一封確認信，包含所有營隊的付款按鈕
           - 使用已修正的 HTML 模板（移除 first()）
```

#### 3-3. Make.com 公式重點

**Iterator 輸入：**
```
get(map(1.data.fields; "value"; "label"; "報名營隊"); 1)
```
這會取得所有選取的營隊 ID 陣列，Iterator 會逐一遍歷。

**Contact 的 interested_camp 寫入：**
```
{{join(map(get(map(1.data.fields; "options"; "label"; "報名營隊"); 1); "text"; "id"; get(map(1.data.fields; "value"; "label"; "報名營隊"); 1)); ";")}}
```
注意：分隔符號為 `;`（分號），這是 HubSpot enumeration 多值寫入的標準格式。

**Deal 的 dealname 命名：**
```
團報_{{孩子姓名}}_{{Iterator 當前營隊的簡稱}}
```

### 關鍵技術發現（Make IML 隱藏 syntax）

除錯過程中發現多項 Make 官方文件未明確記載的 IML 行為，影響後續所有自動化開發。整理如下：

#### 1. Sheet 欄位必須使用 backtick + numeric name

透過 UI 拖拉 Google Sheets 模組欄位產生的 IML，並非用 column letter 而是用 numeric index 包 backtick：

| Sheet UI 顯示 | 實際 IML | 對應內容 |
|---|---|---|
| A | `` 8.`0` `` | 營隊名稱 |
| D | `` 8.`3` `` | 早鳥截止日 |
| E | `` 8.`4` `` | 早鳥團報3人價 |
| G | `` 8.`6` `` | 正常團報3人價 |
| I | `` 8.`8` `` | 早鳥連結 |
| K | `` 8.`10` `` | 正常連結 |

手寫 `8.D`、`8.0`、`get(8;"3")`、`8."A"` 等 syntax 全部 evaluate 失敗，回傳 empty 或字面字串「8」。

#### 2. UI 拖拉 token 帶有 hidden metadata

雖然 IML 字串看起來相同，UI 拖拉產生的 token 內含 Make 內部 metadata，API 直接寫入相同字串的 IML 會 evaluate 失敗或在執行時被截斷。對於複雜 string concatenation 尤其明顯（含多個 `&` 拼接 + 大量 HTML literal + 跨模組 reference）。

#### 3. 日期比較必須 parseDate 強制轉型

直接 `now <= 8.D` 字串比較會失敗（日期 format 不一致導致 `<=` evaluate 為 false）。穩定寫法：

```
{{if(now <= parseDate(8.`3`; "YYYY-MM-DD"); "early_bird"; "normal")}}
```

#### 4. Form fields 用 label match 取代 hardcoded index

HubSpot Form webhook 的 fields 順序不固定。用 `get(1.data.fields[].value; 2)` 會抓錯欄位（拿到家長姓名而非 Email）。穩健寫法：

```
{{get(map(1.data.fields; "value"; "label"; "Email"); 1)}}
```

對 MULTIPLE_CHOICE 欄位（value 是 UUID array），需用 options 解碼成 text：

```
{{get(map(get(map(1.data.fields; "options"; "label"; "團報人數"); 1); "text"; "id"; <selected_uuid>); 1)}}
```

#### 5. Array Aggregator 必須 UI 勾選 aggregated fields

Module 14 (Array Aggregator) 的 mapper 中即使 API 寫了 mapper.payment_button_html，若 UI 「Aggregated fields」dropdown 沒勾選，下游模組無法正確存取 array 內容。

#### 6. SetVariables 不允許自我參照

util:SetVariables 模組內變數不能引用同模組其他變數（如 Module 10.payment_button_html 不能用 `& 10.selected_price &`），會觸發 `BlueprintValidationError`。解法：inline 重寫邏輯，或拆到下游另一個 SetVariables 模組（跨模組 ref 合法）。

#### 限制 3：早鳥日期判斷依賴 sheet column D 格式
