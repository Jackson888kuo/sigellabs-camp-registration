## 新功能規劃

### 功能一：活動設定表（Google Sheets）

**需求背景：** 每次夏令營、冬令營均設有早鳥優惠期間與最後報名繳費日，不同期間對應不同價格；各營隊亦有獨立的付款連結。目前這些資訊須由工程師手動調整 Make.com 流程，缺乏讓行銷人員自助設定的機制。

**採用方案：** Google Sheets 獨立設定表（方案 A）

**設計原則：**
- 行銷人員直接在 Google Sheets 填寫，不須接觸 Make.com 或 HubSpot
- Make.com 在每次收到 Tally 報名時，先讀取設定表，再判斷適用價格與付款連結

**設定表欄位結構：**

| 欄位 | 說明 | 範例 |
|------|------|------|
| 活動名稱 | 本次活動識別名稱 | 2026 夏令營 |
| 早鳥開始日 | 早鳥優惠起始日期 | 2026-03-01 |
| 早鳥結束日 | 早鳥優惠截止日期 | 2026-04-30 |
| 正常期結束日 | 最後報名繳費截止日 | 2026-06-15 |
| 早鳥價格 | 早鳥期間每個營隊費用 | 8500 |
| 正常價格 | 正常期間每個營隊費用 | 9500 |
| 營隊A 付款連結 | 各營隊專屬付款 URL | https://... |
| 營隊B 付款連結 | 各營隊專屬付款 URL | https://... |
| 營隊C 付款連結 | 各營隊專屬付款 URL | https://... |
| （依實際營隊數擴充） | | |

**Make.com 流程異動：**
- 在 Tally Webhook 之後、Create Contact 之前，新增「讀取 Google Sheets 設定表」模組
- 依報名時間與早鳥/正常期日期比對，決定寫入 Deal 的 amount 值
- 依報名的營隊名稱，取得對應的付款連結，注入確認信 HTML

---

### 功能二：追蹤表依活動分 Tab

**需求背景：** 目前追蹤報名表所有報名資料混在同一張工作表，難以按夏令營/冬令營分開追蹤。

**採用方案：** 預先建好各活動 Tab，Make.com 依活動名稱寫入對應 Tab（方案 A）

**設計原則：**
- 每次夏令營/冬令營對應一個獨立 Tab（如「2026夏令營」、「2026冬令營」）
- Tab 由人工預先建立，命名需與設定表的「活動名稱」欄位一致
- Make.com 讀取設定表的活動名稱後，動態指定寫入目標 Tab

**Make.com 流程異動：**
- Google Sheets 寫入模組的「工作表」欄位，改為動態對應設定表中的活動名稱
- 預先建立各活動 Tab 為人工作業，無需自動化

- [ ] 建立 Google Sheets 活動設定表，填入欄位結構

## 風險與注意事項（新增）

| 風險 | 影響 | 緩解措施 |
|------|------|---------|
| 設定表日期格式不一致 | Make.com 日期比對失敗，價格判斷錯誤 | 統一格式為 YYYY-MM-DD，並在 Make.com 加入格式驗證 |
| 活動名稱拼寫不一致 | 設定表與 Tab 名稱不匹配，資料寫入失敗 | 建立命名規範，Tab 名稱與設定表活動名稱欄位需完全一致 |
| 設定表未填寫即收到報名 | Make.com 讀取空值，流程報錯 | 加入錯誤處理模組，讀取失敗時通知管理員 |
