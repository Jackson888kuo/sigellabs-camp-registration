# 太陽實驗室 2026 夏令營團報系統改善企劃書 v1.4

## 專案背景

目前太陽實驗室的團報流程為：Tally 表單 → Make.com 自動化 → HubSpot CRM + Google Sheets 追蹤表 + SendGrid 確認信。

---

## 推薦方案：「一個營隊 = 一筆 Deal」Iterator 架構

**設計原則：**
- 一份報名單、一位家長、一位小孩：對應一個 Contact
- 每個報名的營隊：對應一筆獨立的 Deal，關聯到該 Contact
- 追蹤報名表：自然達成「一列 = 一學生 + 一營隊」
- 確認信：一封信包含所有營隊的付款按鈕

## Phase 5：活動設定表建置 — 功能一（規格確認，待開發）

### 需求背景

每次夏令營、冬令營各梯次均設有早鳥優惠截止日，不同期間搭配不同人數（3人/8人團報）對應不同單價；各梯次亦有獨立的團報付款連結。

目前這些資訊須由工程師手動調整 Make.com 流程，缺乏讓行銷人員自助設定的機制。

### 架構決策：混合架構（WooCommerce API + Google Sheets）

| 資料來源 | 取得什麼 | 原因 |
|---------|---------|------|
| WooCommerce API | 早鳥截止日（date_on_sale_to） | WooCommerce 後台已維護，不重複輸入 |
| Google Sheets 設定表 | 4個團報單價 + 付款連結 | 團報專屬資料，WooCommerce 沒有 |

### WooCommerce 產品結構（已確認）

- **Product** = 一個營隊（variable product）
- **Variation** = 一個梯次（如 [台北] 8/3-8/7）
- 每個 Variation 有：`regular_price`（正常個人價）、`sale_price`（早鳥個人價）、`date_on_sale_to`（早鳥截止日）
- API 端點：https://sigellabs.com/wp-json/wc/v3/products

### Google Sheets 設定表欄位（每梯次一列）

| 欄位 | 說明 | 範例 |
|------|------|------|
| 營隊名稱 | 對應 Tally 選項文字 | 重返案發現場夏令營 |
| 梯次 | WooCommerce variation attribute | [台北] 8/3-8/7 |
| 早鳥 3人團報單價 | 早鳥期間、3人揪團，每人單價 | 10,500 |
| 早鳥 8人團報單價 | 早鳥期間、8人揪團，每人單價 | 9,800 |
| 正常期 3人團報單價 | 正常期間、3人揪團，每人單價 | 12,500 |
| 正常期 8人團報單價 | 正常期間、8人揪團，每人單價 | 11,000 |
| 付款連結 | 該梯次專屬團報付款連結 | https://... |

### Tally 表單新增

新增選擇題「團報人數」，選項：「3人團報」／「8人團報」

（主揪者事先告知 SIGE Labs 人數，取得報名表後每個成員自行填寫、各自付款）

### Make.com 判斷邏輯（2×2 價格矩陣）

```
步驟一：呼叫 WooCommerce API
  → 依營隊名稱 + 梯次查詢對應 Variation
  → 取得 date_on_sale_to（早鳥截止日）

步驟二：判斷期間
  → 報名時間 ≤ date_on_sale_to → 早鳥期
  → 報名時間 > date_on_sale_to → 正常期

步驟三：讀取 Tally 團報人數
  → 3人團報 or 8人團報

步驟四：查詢 Google Sheets 設定表（2×2 矩陣）
  → 早鳥 × 3人 → 早鳥3人團報單價
  → 早鳥 × 8人 → 早鳥8人團報單價
  → 正常 × 3人 → 正常期3人團報單價
  → 正常 × 8人 → 正常期8人團報單價

步驟五：
  → 寫入 HubSpot Deal 的 amount 欄位
  → 取得付款連結，注入 SendGrid 確認信 HTML
```

### 各報名者計費方式
- 每人各自填報名單、各自付款
- Deal amount = 該人應付之團報單價（非總金額）

## Phase 6：追蹤表依活動分 Tab — 功能二（規格確認，待開發）

### 需求背景
目前追蹤報名表所有報名資料混在同一張工作表，難以按夏令營/冬令營分開追蹤。

### 採用方案：預先建好各活動 Tab，Make.com 依活動名稱寫入對應 Tab

**設計原則：**
- 每次夏令營/冬令營對應一個獨立 Tab（如「2026夏令營」、「2026冬令營」）
- Tab 由人工預先建立，命名需與設定表的「活動名稱」欄位一致
- Make.com 讀取設定表的活動名稱後，動態指定寫入目標 Tab

## 風險與注意事項

| 風險 | 影響 | 緩解措施 |
|------|------|---------|
| Tally 多選欄位 value 與 HubSpot option value 不一致 | 寫入失敗 | Phase 3 逐一比對 |
| Make.com Iterator 內發信 | 多封重複確認信 | SendGrid 模組放在 Iterator 之外 |
| 現有 Contact 被覆蓋 | 歷史資料遺失 | HubSpot Update 模式，僅更新指定欄位 |
| WooCommerce API 連線失敗 | 無法取得早鳥截止日，流程中斷 | 加入錯誤處理，失敗時通知管理員 |
| 營隊名稱/梯次拼寫不一致 | Sheets 查詢不到對應列，單價為空 | 建立對照表，確保 Tally 選項與 Sheets 欄位完全一致 |
| 設定表未填寫即收到報名 | Make.com 讀取空值，流程報錯 | 加入錯誤處理，讀取失敗時通知管理員 |
| 活動名稱與 Tab 名稱不一致 | 資料寫入錯誤 Tab | 建立命名規範，兩者需完全一致 |
